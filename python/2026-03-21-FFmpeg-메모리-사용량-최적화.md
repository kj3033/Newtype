### Context
최근 FFmpeg 버전 업데이트 이후, 기존의 리소스 효율성이 떨어지고 메모리 사용량이 급증하여 경량 서버(Low-resource server)에서 OOM(Out of Memory) 현상이 발생하는 문제를 겪었습니다. FFmpeg는 기본적으로 고성능 처리를 위해 버퍼를 크게 잡는 경향이 있어, 제한된 환경에서는 이를 직접 제어해야 합니다.

### Core
메모리 사용량을 제한하고 최적화하기 위해 다음 파라미터를 적용합니다.

* `-threads`: 스레드 수를 명시적으로 제한하여 병렬 처리 시 발생하는 메모리 점유를 제어합니다.
* `-bufsize` 및 `-maxrate`: 비디오 인코딩 시 버퍼 크기를 낮게 설정하여 메모리 적재량을 줄입니다.
* `-preset`: 인코딩 속도와 압축 효율 사이의 균형을 맞춥니다. 'ultrafast'나 'superfast'는 CPU 점유율은 높일 수 있으나 메모리 효율성을 고려할 때 적절한 선택이 될 수 있습니다.

```bash
# 메모리 제한을 위한 FFmpeg 권장 명령어 예시
ffmpeg -i input.mp4 \
  -threads 2 \
  -c:v libx264 -preset veryfast \
  -bufsize 500k -maxrate 1M \
  output.mp4
```

### Insight
FFmpeg의 버전업은 대개 새로운 코덱 지원과 인코딩 품질 향상을 목표로 하지만, 이로 인해 기본 버퍼 할당 정책이 변경될 수 있습니다. 경량 서버에서는 `-threads`를 CPU 코어 수보다 낮게 설정하여 메모리 경합을 줄이는 것이 효과적입니다. 또한, 실시간 처리가 아니라면 배치(Batch) 프로세싱을 도입하고 대기열 관리자(Queue manager)를 통해 동시 실행되는 프로세스 수를 물리적으로 제한해야 안정적인 운영이 가능합니다.
